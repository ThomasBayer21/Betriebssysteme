Einführung & Grundlagen

Was ist ein Betriebssystem?

Ein Betriebssystem (BS) ist die Softwareebene zwischen Anwenderprogrammen und der Hardware. Es verwaltet Ressourcen (CPU, Speicher, E/A) und stellt Schnittstellen bereit.

Aufgaben

  • Prozessverwaltung
  • Speicherverwaltung
  • Dateiverwaltung
  • E/A-Verwaltung
  • Netzwerkdienste

Eigenschaften

  • Ressourcenzuteilung
  • Schutz der Prozesse
  • Abstraktion der HW
  • Virtuelle Maschine

Komponenten (HW)

  • CPU (ALU + Register)
  • Hauptspeicher (RAM)
  • Externer Speicher
  • E/A-Geräte

Schalenmodell (Shell-Modell) Jede Schicht bietet der äußeren Schicht eine virtuelle Maschine. Schnittstellen zwischen Schichten: Systemaufrufe (BS→HW) und API (Programm→BS).

FCFS – First Come First Served (nicht-präemptiv)

Prozesse werden in der Reihenfolge des Eintreffens abgearbeitet. Einfach, aber kein Verdrängen möglich → ein langer Job blockiert alle anderen.

Problem: „Konvoi-Effekt" – kurze Jobs müssen auf einen langen warten.
SJF – Shortest Job First (nicht-präemptiv)

Der Prozess mit der kürzesten geschätzten Ausführungszeit erhält die CPU. Optimal in Bezug auf mittlere Wartezeit.

Geschätzte Bedienzeit zum nächsten Zeitpunkt:

pt+1 = α·pt(1-α)⁰ + α·pt-1(1-α)¹ + … + α·p0(1-α)ⁿ
Problem: Verhungern (starvation) möglich – lange Jobs kommen nie dran. Lösung: dynamische Prioritätsanpassung.
Round Robin (RR) – präemptiv

Jeder Prozess bekommt ein Zeitquantum q. Danach wird er verdrängt und ans Ende der Warteschlange gestellt.

q großq klein
Wenig OverheadViele Kontextwechsel
Verhält sich wie FCFSGute Reaktionszeit
Priority Scheduling (PS)

Jeder Prozess hat eine Priorität. Neue Prozesse werden nach Priorität in die Warteschlange eingeordnet. Bei gleicher Priorität z.B. FCFS.

Achtung: Verhungern möglich! → Lösung: dynamische Prioritätsanpassung (aging)
Multi-Level / Multi-Level Feedback Scheduling

Mehrere Warteschlangen mit verschiedenen Prioritäten:

Multi-level feedback: Jobs können bei längerer Wartezeit in eine Warteschlange höherer Ordnung wechseln.

Synchronisation & Kritischer Abschnitt

🔴 Race Condition Zwei Prozesse lesen/schreiben gleichzeitig auf gemeinsame Daten → nicht-deterministisches Ergebnis!

Anforderungen an eine korrekte Lösung für den kritischen Abschnitt:

1. Mutual Exclusion

Nie zwei Prozesse gleichzeitig im kritischen Abschnitt.

2. Keine Annahmen

Keine Annahmen über Geschwindigkeit oder Anzahl der Prozesse/CPUs.

3. Kein Blockieren außen

Kein Prozess blockiert außerhalb seines kritischen Abschnitts andere.

4. Fairness

Jeder wartende Prozess muss irgendwann eintreten dürfen. Kein ewiges Warten!

Fehlgeschlagene naive Lösungen

Versuch (1) – gemeinsame Variable nextp
C – Falsch (1)
// Prozess 1
while (nextp != 1) { /* busy wait */ }
/* kritischer Abschnitt */
nextp = 2;

// FEHLER: Prozess 1 kann den kritischen Abschnitt
// nicht zweimal hintereinander betreten!
Versuch (3) – beide setzen Flag gleichzeitig
C – Falsch (3)
// Prozess 1
nextp1 = TRUE;
while (nextp2 == TRUE) { /* warten */ }
/* kritischer Abschnitt */
nextp1 = FALSE;

// FEHLER: Beide setzen gleichzeitig auf TRUE →
// beide warten für immer (Deadlock!)

Semaphore (Dijkstra 1965)

p(s) – Passieren
Beim Eintritt in den kritischen Abschnitt. Wartet, falls s = 0 (ein anderer Prozess ist drin).
v(s) – Verlassen
Beim Verlassen. Weckt einen wartenden Prozess auf (s++). Auch bekannt als signal / up.
✔ Atomare Operationen p() und v() sind unteilbar (atomar) – kein Kontextwechsel mittendrin möglich!

Erzeuger-Verbraucher-Problem

Drei Semaphore: mutex (wechselseitiger Ausschluss), frei (freie Pufferplätze), belegt (belegte Plätze).

C – Erzeuger / Verbraucher
/* Initialisierung */
sem_t mutex = 1, frei = N, belegt = 0;

/* Erzeuger */
for (;;) {
    produce(&item);
    p(frei);          // warte auf freien Platz
    p(mutex);         // kritischen Abschnitt betreten
    putin_buffer(item);
    v(mutex);         // kritischen Abschnitt verlassen
    v(belegt);        // signalisiere belegten Platz
}

/* Verbraucher */
for (;;) {
    p(belegt);        // warte auf belegten Platz
    p(mutex);
    getfrom_buffer(&item);
    v(mutex);
    v(frei);          // signalisiere freien Platz
    consume(item);
}
🔴 Deadlock-Gefahr beim Erzeuger-Verbraucher! Falls irrtümlicherweise frei = belegt = 0: Beide Prozesse schlafen für immer → Verklemmung!

Verklemmungen (Deadlocks)

Definition Verklemmung Eine Menge von Prozessen ist verklemmt, wenn jeder Prozess auf ein Ereignis wartet, das nur ein anderer Prozess in der Menge auslösen kann.

Notwendige Bedingungen (alle 4 müssen gleichzeitig gelten – Coffman 1971):

1. Mutual Exclusion

Ressource kann nur von einem Prozess gleichzeitig genutzt werden.

2. Hold and Wait

Prozess hält Ressourcen und wartet auf weitere.

3. No Preemption

Ressourcen können nicht zwangsweise entzogen werden.

4. Circular Wait

Zyklische Wartekette: P1 wartet auf P2, P2 auf P3, … Pn auf P1.

Strategien gegen Deadlocks:

Ignorieren
„Ostrich-Algorithmus" – BS ignoriert das Problem (z.B. frühere Unix-Systeme)
Vermeiden
Banker-Algorithmus: nur sichere Zustände zulassen
Erkennen
Ressourcen-Graphen zyklisch prüfen → dann Prozess abbrechen
Verhindern
Mind. eine Coffman-Bedingung aufheben (z.B. kein Hold-and-Wait)

Systemprogrammierung C

fork / exec / wait / exit

Prozesse erstellen: fork()

C
#include <unistd.h>
#include <sys/types.h>
#include <stdio.h>
#include <sys/wait.h>

int main() {
    pid_t pid = fork();  // Prozess duplizieren

    if (pid < 0) {
        perror("fork fehlgeschlagen");
        return 1;
    }
    else if (pid == 0) {
        /* ═══ KIND-PROZESS (pid == 0) ═══ */
        execlp("ls", "ls", "-l", NULL);  // Programm ersetzen
        exit(1);   // nur wenn exec scheitert
    }
    else {
        /* ═══ VATER-PROZESS (pid = pid des Kindes) ═══ */
        waitpid(pid, NULL, 0);   // warte auf Kind
        exit(0);
    }
}
fork() < 0
Fehler – konnte keinen Kindprozess erstellen
fork() == 0
Wir sind im Kind-Prozess
fork() > 0
Wir sind im Vater, Wert = PID des Kindes
Alle Unix-Prozesse stammen von init (PID 1) ab! fork() erstellt eine exakte Kopie. exec() ersetzt das Programm im Prozess.

POSIX Threads (pthreads)

C – Threads + Mutex
#include <pthread.h>
#include <stdio.h>

pthread_mutex_t mtx = PTHREAD_MUTEX_INITIALIZER;
int counter = 0;

void* thread_func(void* arg) {
    pthread_mutex_lock(&mtx);   // p(mutex)
    counter++;
    pthread_mutex_unlock(&mtx); // v(mutex)
    return NULL;
}

int main() {
    pthread_t t1, t2;
    pthread_create(&t1, NULL, thread_func, NULL);
    pthread_create(&t2, NULL, thread_func, NULL);
    pthread_join(t1, NULL);  // warte auf t1
    pthread_join(t2, NULL);
    printf("counter = %d\n", counter);
    return 0;
}
// Kompilieren: gcc -o prog prog.c -lpthread
FunktionBedeutung
pthread_create(&tid, attr, fn, arg)Thread erstellen
pthread_join(tid, retval)Auf Thread warten (wie waitpid)
pthread_exit(val)Thread beenden
pthread_mutex_lock(&m)Mutex sperren (p-Operation)
pthread_mutex_unlock(&m)Mutex freigeben (v-Operation)

POSIX Semaphore

C – POSIX Semaphore
#include <semaphore.h>
#include <pthread.h>

sem_t sem_frei, sem_belegt, sem_mutex;

void init() {
    sem_init(&sem_mutex,  0, 1);  // 1 = frei
    sem_init(&sem_frei,   0, N);  // N Pufferplätze
    sem_init(&sem_belegt, 0, 0);  // 0 belegt
}

/* p(s)  →  sem_wait  */
sem_wait(&sem_mutex);   // blockiert wenn sem == 0

/* v(s)  →  sem_post  */
sem_post(&sem_mutex);   // erhöht sem, weckt Wartenden

/* Semaphore löschen */
sem_destroy(&sem_mutex);

// Kompilieren: gcc -o prog prog.c -lpthread
sem_init(s, 0, n)
Initialisiere mit Wert n. 2. Param 0 = prozesslokal
sem_wait(s)
p-Operation: s-- (blockiert wenn s==0)
sem_post(s)
v-Operation: s++ (weckt wartenden Thread)
sem_destroy(s)
Semaphore freigeben

Coffman-Bedingungen prüfen (Deadlock)

Für eine Klausuraufgabe: Prüfe systematisch alle 4 Coffman-Bedingungen:

  • ✅ Mutual Exclusion: Kann Ressource exklusiv gehalten werden?
  • ✅ Hold & Wait: Hält Prozess eine Ressource und wartet auf weitere?
  • ✅ No Preemption: Kann Ressource nicht zwangsweise entzogen werden?
  • ✅ Circular Wait: Gibt es einen Kreis in der Warte-Graph?

Sind alle 4 erfüllt → Deadlock möglich. Zerstöre mind. 1 Bedingung zur Vermeidung.
